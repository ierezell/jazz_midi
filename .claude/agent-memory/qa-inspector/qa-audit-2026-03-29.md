## Bugs Found

### BUG 1 — BLOCKER: UserStatsService.getTypeStats throws for hand_independence and dexterity

**File:** `src/lib/UserStatsService.ts:536-550`

`getTypeStats()` has a switch with cases only for 'chord', 'scale', 'progression', 'partition', 'rhythm'.
It hits `default: throw new Error(...)` for 'hand_independence' and 'dexterity'.

`ExerciseResult.exerciseType` (types.ts:47) includes `'hand_independence' | 'dexterity'`.
`BaseExercise.svelte` (line 416-421) passes the exerciseType through a cast but at runtime it's still the original string.

**Impact:** Completing a dexterity or hand_independence exercise throws an unhandled error inside `recordExerciseResult()`. Stats are NOT saved. If in journey mode, the LessonCompleteModal never shows because the error is thrown before `showCompleteModal = true`.

**Fix:** Add cases to `getTypeStats()` for 'hand_independence' → chordStats (or add dedicated stats fields), and 'dexterity' → chordStats (or dedicated).

### BUG 2 — BLOCKER: Lick exercise ignores ?lickId URL param from JourneyService

**File:** `src/routes/exercises/licks/+page.svelte` (no `page.url.searchParams` usage)

JourneyService (lines 337-355) sets lesson params like `{ lickId: 'blues-walking-bass' }` and builds URLs like `/exercises/licks?lickId=blues-walking-bass&unitId=...&lessonId=...`.

The lick exercise page never reads `lickId` from the URL. On `onMount()` it calls `selectRandomLick()` which ignores the param entirely. The user is shown a random lick instead of the one required by the lesson.

**Impact:** All 9+ journey lick lessons (l3-blues-lick, l3-pentatonic-blues-riff, l4-stride-lick, etc.) present the wrong content. Journey completion is effectively random.

**Fix:** In `onMount`, read `page.url.searchParams.get('lickId')`. If present, call `selectSpecificLick(lickId)` instead of `selectRandomLick()`.

### BUG 3 — WARNING: hand_independence page only implements levels 1-2 but journey has levels 3-4

**Files:** `src/routes/exercises/hand_independence/+page.svelte`, `src/lib/JourneyService.ts`

JourneyService lessons:

- `hand-independence-3` (level-5): `params: { level: '3' }`
- `l6-hand-independence-4` (level-6): `params: { level: '4' }`

The page component only has level 1 (L1_LH/L1_RH) and level 2 (L2_LH/L2_RH). The `level` URL param is never read. Both lesson 3 and 4 silently default to level 1.

**Fix:** Either implement levels 3 and 4 in the page, or update JourneyService to use levels 1 and 2 instead.

### BUG 4 — WARNING: Boogie exercise never completes — journey progress for l4-boogie never saved

**File:** `src/routes/exercises/boogie/+page.svelte:139-155`

`isCompleted()` always returns `false`. The exercise advances beats via `setTimeout` inside `isCompleted` as a side effect, but never returns `true`. Therefore `BaseExercise.onCompleteExercise()` is never called, which means `journeyService.completeLesson()` is never called.

The `onComplete` prop is passed `onComplete ?? (() => {})` — a no-op. Even if `isCompleted` returned true, the journey would not update unless the lesson was opened from the journey (which it can be, via `isJourneyMode`).

**Fix:** Return `true` from `isCompleted` when the pattern has been played through a sufficient number of full cycles (e.g. 2 complete cycles).

### BUG 5 — WARNING: Note names exercise never fires journey completion

**File:** `src/routes/exercises/names/+page.svelte:162-169`

`isNoteNameCompleted()` always returns `false` (continuous practice mode). For journey lesson `l0-notes`, the exercise never signals completion, so `journeyService.completeLesson()` is never called via BaseExercise. The user would need to play notes forever without any journey feedback.

Additionally, no `exerciseType` prop is passed to BaseExercise, defaulting to 'chord'. Note practice stats are recorded under chord stats.

**Fix:** Define a completion threshold (e.g., 10 correct identifications in a row). Return `true` from `isNoteNameCompleted` when reached.

### BUG 6 — WARNING: getNoteRole returns 'unknown' for minor thirds and minor sevenths

**File:** `src/lib/MusicTheoryUtils.ts:335-357`

`getNoteRole` only maps `difference === 4` to 'third' (major third) and `difference === 11` to 'seventh' (major seventh). Minor thirds (3 semitones) and minor sevenths (10 semitones) return 'unknown'.

This means the keyboard visual highlighting (note role coloring) is wrong for all minor chords, half-diminished chords, and dominant 7th chords. The third and seventh notes show as 'unknown' instead of their role.

**Fix:** Add cases for `difference === 3` → 'third', `difference === 10` → 'seventh'. Also `difference === 6` → 'fifth' (flat five) for dim/half-dim.

### BUG 7 — WARNING: Accuracy formula broken for sequential/lick exercises

**File:** `src/components/BaseExercise.svelte:407-410`

```typescript
const accuracy = Math.max(0, Math.round(((expectedCount - mistakes) / expectedCount) * 100));
```

For lick, dexterity, and partition exercises, `expectedNotes` at completion time contains only 1 note (the next-note-to-play model). So `expectedCount = 1`. One wrong note gives `accuracy = max(0, (1-1)/1*100) = 0%`.

A user who plays a 20-note lick perfectly except for 1 fumbled note gets 0% accuracy and 1 star instead of the ~95% / 3-star result they deserve.

**Fix:** Track total notes played and total mistakes across the full exercise lifetime, then compute accuracy as `(totalPlayed - totalMistakes) / totalPlayed * 100` at completion time.
